## 1. What I understood the scope to be

Moving RamenHire from "no real user auth, admin reviews everything by hand" to "companies have real Supabase Auth accounts and self-service ownership of their own data," while closing the specific deferred finding from `security-audits/2026-07-10-audit.md` (1.1/1.2 — RLS treats `authenticated` as a proxy for "admin," which was only safe because no real end-user auth existed yet).

In build order, as given:

1. **Auth** — Supabase Auth, email/password (not magic-link). Sign-in page. Forgot-password via emailed magic link. Rate limiting on login attempts specifically, separate bucket from the existing general-form rate limiting.
2. **Header UI** — logged-out: "Sign In" button. Logged-in: avatar → dropdown (Account/Company Settings, Sign Out). "Post a Job" always visible; routes to registration/sign-in when logged out, to the post-job form (gated by verification status) when logged in.
3. **Registration** (update, not rebuild) — a real Auth account is created at registration time, not after approval. The existing email-verification step (token-gated entry into the admin queue) is unchanged. A company can log in anytime post-registration to see their own status. Can't publish a live listing until admin-approved. **Unverified-account retention:** 15-day window for not-yet-approved accounts, with pending-status messaging pointing to hello@ramenhire.com; after 15 days, eligible for removal via the same purge mechanism as section 7, as a distinct case from the 90-day approved-account window.
4. **Post-job flow** — `/post-job` gated on auth + verification status (not logged in → redirect; logged in + approved → form; logged in + unverified → pending-status view). Approved companies get full self-service create/edit/delete/publish, no more per-post admin review. Draft-for-unverified-companies is explicitly deferred to a future sprint (after a dashboard exists) — not building it now. Trim `/post-job`: drop fields that now live only on the company profile (company name, website, description, why-work-here, revenue range); keep listing-only fields.
5. **Company page self-service** — create/edit/delete of a company's own listings lives on the company profile page for now (no separate dashboard yet). Strict per-company isolation — never another company's listings.
6. **Account/Company Settings page** — reachable from the avatar dropdown. Contact email management. Self-service delete/data request, tied to the soft-delete mechanism (fulfills the Privacy Policy's stated rights promise with a real feature).
7. **Soft delete** — new nullable `deleted_at` on both `companies` and `job_listings` (distinct concept from `job_listings.is_active`, which is unrelated — temporary deactivation, no timeline). `NULL` = active, timestamp = deleted/counting down, restore = back to `NULL`. Retention: 30 days (listings), 90 days (companies). Explicitly a UX grace-period feature, not a tax/compliance claim — must not be described as satisfying tax record-retention anywhere, legal pages included. A scheduled job (pg_cron or Vercel Cron hitting an API route) permanently purges rows past their window, cascading cleanly (applications tied to a purged listing, CV files in Storage, so nothing is silently orphaned).
8. **`is_admin` + first admin** — boolean on the company/account table. Grants cross-company access for approval/moderation — a permission, not a public-directory visibility rule (directory visibility stays governed only by approval status + active listings). User will manually flip RamenHire's own company row to `is_admin = true` after this ships; no special-casing needed in code beyond the flag.
9. **Legacy listings** — the ~19 curated listings with no real company account stay as-is. Future adoption by a real company is manual/admin-side, not automated matching. If `job_listings` doesn't already have a real FK to `companies` (it doesn't — confirmed below), add one now regardless, since sections 4–5's ownership model needs it.
10. **RLS overhaul** — replace the broad `authenticated`-role grants across `applications`, `job_post_requests`, `subscribers`, `companies`, `job_listings` with real per-company scoping; `is_admin` gets explicit broader access, not inherited from plain `authenticated`. Updated incrementally as each piece is built, not in one big bang at the end.
11. **Close out the audit finding** — mark 1.1/1.2 resolved in `security-audits/2026-07-10-audit.md` (or `CHAT_CONTEXT.md`), referencing this sprint's real scoping.
12. **End of sprint** — full security audit of the new auth/RLS surface (new dated file). Same pass reviews Privacy Policy + Terms of Use in full (not just the obviously-stale "no accounts"/retention sections) for anything else this sprint makes incorrect. Code review/optimization pass across everything built.

**Hard constraints restated:** no production push or migration at any point until explicitly approved; local Supabase stack only throughout; migrations-first workflow (`supabase/migrations/` → `npx supabase db reset` locally); this file gets updated as a running log, not written once.

## 2. Current-state findings that shape the plan

Verified by reading the actual code/schema before writing anything, not assumed:

- **No FK from `job_listings` to `companies` today.** `20260705000003_create_job_listings.sql` has no `company_id` column at all. Homepage/company-profile matching is 100% heuristic (`lib/companies.ts` → `companyMatchesJob()`, matches by normalized website or exact-lowercased name). Confirms section 9's premise — a real FK is new work, not a rename.
- **`companies` table has no `auth_user_id`, no `is_admin`, no `deleted_at`.** All net-new columns.
- **`job_listings` has no `deleted_at`** (has `is_active`, which is a separate, pre-existing concept per the sprint prompt's own instruction not to conflate them).
- **No `@supabase/ssr` package, no `middleware.ts`, no cookie-based session handling anywhere.** `package.json` only has `@supabase/supabase-js`. `lib/supabase.ts` is a plain browser/anon client; `lib/supabase-admin.ts` is a plain service-role client used only inside Route Handlers and one Server Component. This app has never had a logged-in user before — this sprint is the first thing that needs a real session lifecycle (login persists across requests/pages), which `@supabase/supabase-js` alone doesn't give you in the App Router. This is new infrastructure, not a config tweak.
- **No shared `Header`/`Nav` component.** Every page (`HomeClient.tsx`, `PostJobClient.tsx`, `RegisterClient.tsx`, `companies/[slug]/page.tsx`, `companies/page.tsx` presumably) hand-rolls its own `<nav>` block with duplicated inline styles. Rendering auth state (Sign In vs. avatar dropdown) consistently means this needs to become one shared component — flagged as required infrastructure for section 2, not incidental refactoring.
- **`rate-limit.ts`'s `checkRateLimit(bucketKey, max, windowMinutes)` already generalizes cleanly** to a login bucket (e.g. `login:${ip}` or `login:${ip}:${email}`) — no new rate-limiting mechanism needed, just a new bucket key/limit pair, consistent with "separate bucket" in the sprint prompt.
- **Every existing public write already goes through a server-side Route Handler using the service-role client** (`/api/apply`, `/api/post-job`, `/api/subscribe`, `/api/companies/register`, `/api/companies/confirm-logo`, `/api/companies/resend-verification`, `/api/contact`) — none write directly from the browser with the anon key. Login should follow the same shape for the rate limiting requirement to be real rather than cosmetic (see Open Question 6 and Assumption below).
- **Storage path convention is consistent and reusable for purge cascades:** both `cvs` and `company-logos` use `{owning_row_id}/{timestamp}.{ext}` (`lib/storage-upload.ts` → `buildStoragePath()`). CVs are namespaced by `applications.id`, not by job or company — so purging a job listing means walking its `applications` rows individually for Storage cleanup, not deleting a single prefix.
- **`applications.job_id` → `job_listings.id` has no `ON DELETE` clause** (`20260705000004`), i.e. defaults to `RESTRICT` — a raw `DELETE` on `job_listings` would currently fail if applications reference it. The purge job must delete dependent `applications` rows (and their CV Storage objects) before deleting the parent `job_listings` row — can't rely on a cascade FK alone without deciding that's actually what we want (see design below).
- **Company registration already creates zero rows in `job_post_requests`** — that table is exclusively the pre-company-account "post a job" queue. Sections 4–5 imply approved companies will publish directly into `job_listings` going forward, which would make `job_post_requests` stop receiving new rows entirely post-sprint. Flagged as Open Question 1 rather than assumed silently, since it's a real behavior change.
- **Legal pages (`privacy-policy.md`, `terms-of-use.md`) explicitly state today "there is no account system, no login"** (Privacy §1) and "There is no user account system... submissions are tied to the email address provided, not to a login" (Terms §2) — both flatly wrong after this sprint. Privacy §6 ("no automated deletion or retention policy") and §7 ("no automated self-service tool" for deletion) are also directly contradicted by sections 6–7 of this sprint. Confirms section 12's instruction that this needs a full-section pass, not a patch of just the obvious lines.
- **`docs/ARCHITECTURE.md` and `docs/SECURITY.md` are already stale** independent of this sprint (e.g. ARCHITECTURE.md still says "No backend API routes — all data operations go directly to Supabase from the client," which has been false since the rate-limiting/bot-protection work). Not in this sprint's stated scope (1–12 above don't mention these two files), so I'm leaving them alone unless told otherwise — noting it here so it's a known gap, not a rediscovered one later.

## 3. Key design decisions (proposed — confirm or correct)

These are the concrete technical choices the sprint description leaves to implementation judgment. Building on these unless corrected:

- **Identity model:** add `companies.auth_user_id uuid UNIQUE REFERENCES auth.users(id)`. One Supabase Auth user ↔ one `companies` row. This is the row that gets `is_admin` (per section 8's explicit "company/account table" instruction — not a separate `profiles` table, which was only ever the audit's rough suggestion, superseded by this sprint's own spec).
- **Registration creates the Auth account with `email_confirm: true`** (service-role admin API), so Supabase's own "confirm your email" flow never fires. The existing custom `verification_token`/48h-expiry flow remains the *only* email-verification gate into the admin queue, exactly as instructed — this avoids two parallel "check your email" flows that would confuse the applicant and contradict "stays exactly as it is today."
- **Login goes through a new `/api/auth/login` Route Handler**, not a raw client-side `supabase.auth.signInWithPassword()` call — mirrors the shape of every other write path in this app (rate-limit check first via `checkRateLimit()`, then the real operation) and is what makes "rate limiting specifically on login attempts" an enforced boundary instead of a UI-only speed bump that a direct API call would skip.
- **New infrastructure required for the above:** `@supabase/ssr` (official cookie-session package for Next.js) + `middleware.ts` for session refresh. This is genuinely new to the project (see Current-State Findings) — will vet the package per `docs/SECURITY.md`'s package-checklist before installing, and run `npm audit` after.
- **Shared `components/Header.tsx`** (or similar), reading auth state client-side, replacing the duplicated inline `<nav>` in every page. Renders Sign In / avatar+dropdown, and the always-visible Post-a-Job link with its auth-aware destination.
- **RLS ownership + admin-check pattern:** a `SECURITY DEFINER` SQL function (e.g. `is_admin()`) that looks up the caller's own `companies.is_admin` via `auth.uid()`, used from other tables' policies — avoids the recursive-policy trap of a `companies` RLS policy needing to query `companies` itself under RLS. Will validate this pattern directly against the local stack before leaning on it everywhere (same "verify, don't assume" discipline as the Storage-policy lessons in the 2026-07-08/07-10 incidents).
- **`job_listings.company_id uuid REFERENCES companies(id)`, nullable.** Legacy rows keep `company_id = NULL` and stay exactly as visible/manageable as today (admin-only, via dashboard). New company-created listings always set it.
- **Purge cascade, most-specific first:**
  - Job listing purge: delete each dependent `applications` row's CV Storage object (`{application.id}/...` in `cvs`) → delete the `applications` rows → delete the `job_listings` row.
  - Company purge: run the above for every job listing owned by the company → delete the company's Storage object(s) under `{company_id}/...` in `company-logos` → delete the `companies` row → delete the linked `auth.users` account via the admin API (so no orphaned, still-loginable account outlives its profile).
  - Implemented as an explicit application-level cascade (script/API route logic), not relying on FK `ON DELETE CASCADE` alone, so each step can be logged/verified — consistent with this project's existing "verify, don't trust that DROP/DELETE succeeded silently" discipline from past Storage-policy incidents.
- **Purge scheduling: leaning Vercel Cron Job hitting an API route** over `pg_cron` — no new Postgres extension, testable locally against the Docker stack the same way every other server-side operation in this app already is, and consistent with the existing Route-Handler-centric architecture. Flagged as Open Question 7 since the sprint description offered both.

## 4. Assumptions

Proceeding on these unless corrected in your review:

1. `job_post_requests` stops receiving new rows once approved companies can publish directly to `job_listings`; the table is kept only for its 5 existing legacy rows, with RLS tightened per section 10 for consistency (admin-only access, no per-company ownership since it has none).
2. `subscribers` and `job_post_requests` get scoped to **admin-only** authenticated access (via `is_admin()`), not per-company scoping — neither has a real ownership concept (`subscribers` isn't tied to any company; `job_post_requests` predates company accounts and has no FK to one).
3. Approved companies do **not** gain the ability to view their own listings' applicants in this sprint — no such UI is in scope 1–9, so `applications` RLS will scope to admin-only, matching today's actual (Supabase-dashboard-only) capability, just properly enforced instead of merely assumed.
4. The 15-day unverified-account clock starts at `companies.created_at` (covers time sitting in `unverified` before the applicant even clicks the verification email, not just time spent in `pending` awaiting admin review).
5. No self-service "cancel deletion" / restore UI is required this sprint — recovery within the 30/90-day grace window is a manual, contact-support path, not a button in Settings. (Soft delete itself still works exactly as specified; this is only about whether *undoing* it needs its own UI yet.)
6. Login rate limit: defaulting to a tighter bucket than the general 5/hour form limit, e.g. **10 attempts / 15 minutes, keyed by IP** (with a secondary email-keyed bucket if brute-forcing one specific account turns out to matter more than distributed guessing) — exact numbers easy to change, flagging so you can redirect if you want something different.
7. `docs/ARCHITECTURE.md` / `docs/SECURITY.md` are left untouched this sprint (already stale independent of this work, not named in scope 1–12) — only `CLAUDE.md`, `CHAT_CONTEXT.md`, `security-audits/`, and the legal pages get touched, per the sections that actually name them.

## 7. Implementation plan — step by step

Each step is scoped to be independently completable and independently verifiable, and leaves the app in a working (build-passing, no half-wired dependency on a future step) state. Ordered so infrastructure lands before anything depends on it. Steps within a phase are sequential; phases are sequential.

### Phase A — Foundations (no user-facing auth behavior yet)

**A1. Install `@supabase/ssr`, add server/browser Supabase clients, add `middleware.ts`**
- Vet `@supabase/ssr` against `docs/SECURITY.md`'s package checklist, install, `npm audit`.
- Add `lib/supabase-server.ts` (server-side client reading cookies) alongside the existing `lib/supabase.ts` (browser/anon) and `lib/supabase-admin.ts` (service-role) — no existing file is replaced, this is additive.
- Add `middleware.ts` for session cookie refresh, scoped to avoid interfering with existing Route Handlers.
- **Verify:** `npm run build` passes; a temporary server-side page/log confirms a session cookie set via Supabase Studio's SQL (or a manually-created test user) round-trips through `middleware.ts` without errors; remove the temporary check before moving on.

**A2. Migration: new columns + FK (schema only, no RLS changes yet)**
- New migration file: `companies.auth_user_id uuid UNIQUE REFERENCES auth.users(id)`, `companies.is_admin boolean NOT NULL DEFAULT false`, `companies.deleted_at timestamptz`, `job_listings.deleted_at timestamptz`, `job_listings.company_id uuid REFERENCES companies(id)` (nullable, no `ON DELETE` yet — decided explicitly in A-phase purge design, not left to Postgres default).
- `npx supabase db reset` locally.
- **Verify:** `list_tables`/Studio shows all 5 new columns with correct types/defaults; existing app (homepage, company pages, post-job, registration) still works unmodified — this migration must be a strict no-op for current behavior since nothing reads these columns yet.

**A3. Shared `components/Header.tsx` (static — no auth wiring yet)**
- Extract the duplicated inline `<nav>` from `HomeClient.tsx`, `PostJobClient.tsx`, `RegisterClient.tsx`, `companies/[slug]/page.tsx`, and `companies/page.tsx` into one shared component. Behavior identical to today (no Sign In / avatar yet — that's Phase B).
- **Verify:** every page visually matches its current nav pixel-for-pixel, all existing links still work, no TS/build errors.

### Phase B — Auth core

**B1. Sign-in page + `/api/auth/login` Route Handler + login rate-limit bucket**
- New bucket key (e.g. `login:${ip}`), 10/15min per confirmed Assumption 6, via the existing `checkRateLimit()` — no new rate-limit mechanism.
- Route Handler calls `signInWithPassword` server-side via the Phase A1 server client, sets the session cookie.
- Sign-in page posts to it, handles success/error/rate-limited states.
- **Verify:** manually create one test company Auth user (Studio or a temporary script), confirm login succeeds and session persists across a reload; confirm wrong password fails cleanly; confirm the 11th attempt within 15 minutes is blocked.

**B2. Forgot-password flow**
- Request-reset page → Supabase's built-in reset email (magic link) → reset-password page that sets a new password.
- **Verify end-to-end locally:** local Supabase stack captures outgoing email (Inbucket/Mailpit) — confirm the reset link arrives there, confirm following it lands on a working reset form, confirm the new password logs in via B1's flow.

**B3. Wire `Header.tsx` to real auth state**
- Logged-out: "Sign In". Logged-in: avatar → dropdown (Account/Company Settings link, Sign Out). "Post a Job" always visible, destination branches on auth state (Phase D wires the verification-status branch once that data is available).

### Phase C — Registration integration

**C1. Registration creates a real Auth account**

## 8. Deliberately out of scope (deferred to a future sprint)

- **In-app admin moderation.** Admins cannot edit/delete another company's listing or profile through the website this sprint (D2 is owner-only, no admin-override branch). All moderation (removing a spam listing, fixing something on request, etc.) continues exactly as today: manual, via Supabase Studio with the service-role connection, bypassing the app entirely. A real admin dashboard is deferred — same reasoning as other deferrals in this project: not enough real usage yet to know what it should actually look like. `is_admin`'s RLS-level access (G1) is built now specifically so a future dashboard has a real enforcement layer to sit on top of, but ships with zero app-layer surface this sprint.
- **Draft-post capability for unverified companies** (already deferred in the original scope, section 4) — needs a dashboard to make sense, not building it now.
- **Self-service "cancel deletion" / restore UI** (confirmed Assumption 5) — recovery within the grace window is a manual contact-hello@ path this sprint.

## 9. Note on RLS timing risk (why production stays off-limits through Phase G)

Phases B–F build real login sessions and app-layer ownership checks *before* Phase G's database-level RLS rewrite lands. For that window, ownership enforcement lives only in application code — the database would still accept a broader authenticated grant if something bypassed the app layer. This is specifically why no part of this sprint touches production, not just as a blanket restatement of the standing rule: shipping real company accounts mid-sprint, before Phase G and G4's advisor sweep both pass clean, would be strictly worse than today's state (today, no real end-user sessions exist at all; mid-sprint, real sessions would exist without the DB-level backstop that's supposed to contain them). Production stays untouched until G4 is clean and the full plan is locally verified.
