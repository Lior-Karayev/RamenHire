# Security Audit — 2026-07-21

**Scope:** The auth/RLS surface built in this sprint — Supabase Auth (email/password), session handling, login rate limiting, self-service ownership checks (companies + job_listings), the soft-delete/purge cascade and its cron route, and the `is_admin` flag's actual reach. Does **not** re-cover ground already audited on 2026-07-10 except to confirm those specific findings (1.1/1.2) are now genuinely closed, not just claimed closed.

**Prompted by:** The sprint's own end-of-phase instruction (`sprints/2026-07-21-company-auth-sprint.md`, Section H2) — a full security audit specifically covering what this sprint built, same discipline as the 2026-07-10 audit rather than assuming the sprint's own step-by-step verification substitutes for an independent adversarial pass.

Everything below is **report-only** unless explicitly marked as fixed in this same session. Wait for explicit direction per finding before touching anything not already fixed.

## 1. Auth / Session Handling

### 1.1 — Login rate limiting is IP-only; no secondary account-keyed bucket (Low–Medium)

`login:${ip}`, 10 attempts/15min (Step B1), confirmed working via direct testing during the sprint. But the bucket is keyed only by IP — a distributed attacker (a small botnet, or any setup with multiple egress IPs) could brute-force **one specific known account** indefinitely, since no single IP would ever cross its own 10/15min threshold. This was an explicit, confirmed trade-off going into the sprint (Assumption 6: "no secondary email-keyed bucket needed... unless genuinely trivial to add") — noting it here as the residual risk that trade-off accepted, not as something missed. Adding a second `login:${email}` bucket alongside the IP one would close this at low implementation cost, whenever it's judged worth doing.

### 1.2 — Minimum password length is 6 characters (Low)

Matches the local Supabase config's own floor (`minimum_password_length = 6` in `supabase/config.toml`) and the client/server validation added in Step C1. This is weak by current general-purpose standards (many services default to 8–12). Worth a deliberate decision — raise it before production, or accept it for an MVP-stage B2B product where the bigger risk is probably reused passwords rather than brute-forceable short ones (rate limiting per 1.1 already bounds guessing speed).

### 1.3 — Password-reset only completes in the same browser that requested it (Informational)

Inherent to the PKCE flow (Step B2), not a bug: the `code_verifier` needed to exchange the emailed code for a session is stored in a cookie set on the browser that called `resetPasswordForEmail`. A user who requests a reset on a laptop and opens the email link on their phone will hit the "This link isn't valid" state even though the code itself hasn't expired — confirmed by design during B2's own testing (the flow required a single persistent cookie jar to work end-to-end). Worth a UI copy tweak ("open this link on the same device/browser you requested it from") rather than a code fix — switching away from PKCE to avoid this would trade away PKCE's own security properties for cross-device convenience, not obviously worth it at this stage.

### 1.4 — Orphaned Auth account if the compensating cleanup itself fails (Low, low-probability)

`/api/companies/register` (Step C1) creates the Auth account first, then the `companies` row; if the row insert fails, it calls `supabaseAdmin.auth.admin.deleteUser(authUserId)` to clean up — but doesn't check or log whether *that* call succeeds. If the compensating delete itself fails (transient network blip, momentary Auth service hiccup), the result is a real, loginable Auth account with no `companies` row attached and no self-service recovery path: a retry with the same email fails at the `createUser` step with "already exists" (since `auth_user_id` is `UNIQUE`), and the now-logged-in-but-companyless user's only option on `/account` is "Register a company," which would fail the identical way. Low probability (requires two independent failures in sequence) but a real dead end with no recovery this sprint — flagging for either better logging/alerting on this specific failure mode, or a manual admin-side fix path (delete the orphaned Auth user via Studio) documented somewhere findable.

## 2. RLS / Ownership (Steps G1–G3)

### 2.1 — Findings 1.1/1.2 from 2026-07-10 confirmed genuinely closed, with additional adversarial tests beyond what the sprint steps themselves ran

The sprint's own G1–G4 verification already confirmed the core scoping (own-row/own-company access, cross-company rejection, admin bypass, zero remaining blanket policies). Two additional attack angles were tested specifically for this audit, since they weren't explicitly covered during G2:

- **Ownership-transfer attempt via raw `UPDATE`:** a company tried to `PATCH` its own listing's `company_id` to a different company's id directly against PostgREST (bypassing the app's route, which doesn't accept a client-supplied `company_id` at all — this tests the RLS layer specifically, as a defense-in-depth check). Correctly rejected with a real `42501` row-level-security violation; `company_id` confirmed unchanged in Postgres afterward. This works because `WITH CHECK` evaluates against the *new* row, and the spoofed `company_id` doesn't match the caller's own `my_company_id()`.
- **Legacy listing (`company_id IS NULL`) access via raw RLS:** a real approved company attempted to `UPDATE` a legacy-shaped listing (no owning company) directly against PostgREST. Correctly returned zero rows affected — `NULL = my_company_id()` evaluates to `NULL`/false in Postgres, falling through to `is_admin()`, which a non-admin session doesn't have.

No new findings here — this section exists to record that closure was independently re-tested, not just re-asserted.

## 3. Purge Cascade / Cron Route (Steps F2–F3)

### 3.1 — `CRON_SECRET` comparison was non-constant-time — fixed same session (zero-downside hardening)

`app/api/cron/purge/route.ts` compared the `Authorization` header against the expected value with plain `!==` string inequality. For a random 64-character hex secret, a timing attack to recover it over a real network is impractical, but it's not best practice and the fix has no behavior trade-off — same category as the 2026-07-10 audit's own `x-vercel-forwarded-for` preference fix (identical behavior today, strictly more robust). Now uses Node's `crypto.timingSafeEqual`, with an explicit length-mismatch guard (`timingSafeEqual` throws rather than returning `false` on unequal-length buffers) that itself stays constant-time by comparing the header against itself rather than short-circuiting. **Verified:** rebuilt, re-ran all four cases (no header, wrong secret, wrong secret of a *different* length, correct secret) — all four behave identically to before the fix, confirming no regression.

### 3.2 — No persistent audit trail of what the purge cascade actually deletes (Informational)

The cron route returns a summary count (`companiesPurged`, `jobListingsPurged`, `unverifiedAccountsPurged`) in its JSON response, but nothing is written anywhere persistent — if a company or listing disappears and someone later asks "why," the only trail is whatever Vercel's own function logs happen to retain. For a system doing genuinely permanent, irreversible deletion, consider logging each purge decision (company/listing id, reason, timestamp) to a small durable table or external log sink before this ships to production. Not fixed this sprint — a design decision, not a quick patch.

### 3.3 — Purge failures surface only via `console.error`, no alerting (Informational)

If `runSoftDeletePurge()`/`runUnverifiedAccountPurge()` throw, the route logs and returns a 500 — Vercel's own Cron dashboard does track failed invocations, which may already be sufficient monitoring, but this hasn't been confirmed against an actual Vercel deployment (this sprint never left the local stack). Worth a quick confirmation once this sprint reaches a real Vercel environment, before relying on it.

## 4. `is_admin` Flag Reach (Steps G1, G4)

Confirmed exactly as scoped in Section 8 of the sprint file, not a finding: `is_admin()` has real teeth at the RLS layer (confirmed empirically throughout G1–G3) but **zero application-layer surface** — no route or page checks or exposes it. The only way to set it is direct SQL/Studio access, exactly as the sprint intended ("register RamenHire itself as a company and manually set its `is_admin` to true after ship — no special-case code needed"). Recorded here as confirmed-by-design, not overlooked.

## 5. Carried over from the 2026-07-10 audit, still genuinely open (not touched this sprint)

Both surfaced incidentally during Step G4's grants sweep (running `get_advisors` against production as informational context); neither is part of this sprint's stated 5-table RLS scope, so neither was fixed here — flagging for a deliberate decision rather than silently deferring again:

- **`company-logos` Storage bucket** — `public_read_company_logos` allows listing the entire bucket's contents, not just fetching a known path. Unrelated to the auth/RLS work this sprint did; the original 2026-07-10 audit's finding 2.2 already noted a related inconsistency in this same policy.
- **`public.rls_auto_enable()`** — a pre-existing `SECURITY DEFINER` function (not created by this sprint) callable via RPC by both `anon` and `authenticated`. Worth a look to confirm it doesn't do anything a non-admin caller shouldn't be able to trigger.

## Summary

This sprint's core claim — that findings 1.1/1.2 from 2026-07-10 are closed — held up under independent re-testing, including two attack angles (ownership-transfer-via-UPDATE, legacy-row access) that weren't explicitly part of the sprint's own step-by-step verification. One small, zero-downside hardening was applied and verified in this same session (constant-time `CRON_SECRET` comparison). No critical or high-severity findings. Four Low/Low-Medium findings worth a deliberate decision before production (IP-only login rate limiting, 6-character password minimum, the cross-device password-reset limitation, and the orphaned-Auth-account edge case), three Informational notes about the purge cascade's operability once it's actually running against a real schedule, and two pre-existing, unrelated findings carried over for a future pass. Nothing found here blocks moving forward with Phase H's remaining steps, but several of these are worth resolving — or consciously accepting — before any of this sprint's migrations reach production.

## Verified — 2026-07-24 (pre-production final check)

Re-checked findings 1.1–1.4 and 3.1–3.3 against the current code, as part of a final pre-production review pass (not a new audit — see `sprints/2026-07-21-company-auth-sprint.md` progress log for the same date). All eight re-verified directly, not re-asserted from this file's own text:

- **1.1 (IP-only login rate limiting) — now resolved, not just an accepted trade-off.** `app/api/auth/login/route.ts` now checks a second bucket, `login:email:${email}`, alongside the original `login:${ip}` — both 10/15min — closing exactly the distributed-attacker gap this finding described. (Not present when this audit was originally written; added since, undated in this file until now.)
- **1.2 (6-character password minimum) — now resolved.** `supabase/config.toml`'s `minimum_password_length` is `8`, and both `app/api/companies/register/route.ts` and `app/api/auth/reset-password/route.ts` independently enforce `length < 8` rejection — config and both app-level checks now agree at 8, not 6.
- **1.3 (cross-device password reset) — mitigated via the suggested copy fix.** `ForgotPasswordClient.tsx`'s success state now reads "Open the link on the same browser and device you requested it from — it won't work elsewhere," exactly the UI-copy mitigation this finding proposed. The underlying PKCE same-browser constraint is unchanged (expected — that was never in question), but the silent-dead-end UX gap is closed.
- **1.4 (orphaned Auth account on double-failure) — mitigated.** `cleanupOrphanedAuthUser()` in `app/api/companies/register/route.ts` now checks the delete call's own `error` and `console.error`s a specific, actionable message (account id, manual Studio recovery instructions) if the compensating cleanup itself fails. The underlying low-probability dead end still has no automated recovery (never claimed it would), but it's no longer silent.
- **3.1 (constant-time `CRON_SECRET` comparison)** — confirmed still in place (`timingSafeEqual` in `app/api/cron/purge/route.ts`), re-tested live: missing header and wrong secret both correctly 401. No regression.
- **3.2 (no persistent purge audit trail)** — confirmed still open, as documented; still a design decision, not revisited this pass.
- **3.3 (purge failures only via `console.error`)** — confirmed still open, as documented; still pending a real Vercel deployment to confirm Cron-dashboard visibility is sufficient.
- **RLS closure (Section 2.1)** — independently re-queried `pg_policies` directly against the local database (not re-derived from the app): 12 policies across the 5 touched tables, zero blanket `true` conditions anywhere. Matches this file's original claim exactly.

Net effect: four of the eight items this file flagged as open (1.1, 1.2, 1.3, 1.4) have since been resolved or mitigated by later local work not captured in this file until now. The three purge-related Informational items (3.1 fixed, 3.2/3.3 open) and the RLS closure claim are unchanged and re-confirmed. Nothing new found. No regressions.
