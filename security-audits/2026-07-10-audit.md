# Security Audit — 2026-07-10

**Scope:** Full RLS policy review, Storage bucket policies, API routes, service-role client isolation, and env var handling across RamenHire.

**Prompted by:** Community feedback from Dev.to commenter **Pon**, whose two prior comments each caught a real production issue (the `cvs` bucket anon-upload signed-URL gap, and questions around orphan-file handling). This is the third round of feedback from the same thread. Two concrete items from that round were fixed and verified in this same session (see below); this document covers the broader audit requested alongside them.

**Companion fixes (same session, already applied and tested locally):**
1. Removed the `authenticated`-role DELETE grant on the `cvs` Storage bucket (`supabase/migrations/20260710000009_revoke_authenticated_delete_cvs.sql`). Verified via direct Storage API calls with a crafted `role: authenticated` JWT — confirmed 403 after the fix, and confirmed 200 (successful delete) with the same JWT before the fix, proving the test isolates the right behavior.
2. Verified — by deliberately forcing DB insert failures against the running local dev server, not just reading the code — that `/api/apply` and `/api/companies/register` never return a usable signed upload URL unless the corresponding row insert actually succeeded. No fix was needed here; the existing gating (mint URL → attempt insert → only include `upload` in the response if insert succeeded) already closes the orphan-file path. Confirmed no orphan rows were created and no `upload` key appeared in either failure response.

Everything below is **report-only** — nothing in this section has been changed. Wait for explicit go-ahead per item before touching code.

## 1. RLS Policies

### 1.1 — "authenticated = admin" is an unenforced assumption, repeated across 5 tables (Medium)

`applications`, `job_post_requests`, `subscribers`, `companies`, and `job_listings` all grant the Postgres `authenticated` role broad SELECT/UPDATE/DELETE (and for `job_listings`, INSERT too — the broadest grant in the schema) via `USING (true)` policies with no ownership or role-level scoping. This is the same pattern already flagged as fragile for the `cvs` bucket DELETE grant fixed in this session.

**Why it's contained today:** there is no real end-user auth anywhere in the app (per `CLAUDE.md`: "no auth for job-seekers, admin via Supabase dashboard only"). The only way to obtain a JWT with `role: authenticated` is via the Supabase dashboard itself or by signing one directly with the project's JWT secret — a genuinely secret value, distinct from the anon/service-role keys, that has not leaked (checked git history — clean).

**Why it's still worth closing:** the moment any future feature introduces real candidate/employer accounts (even something as simple as a magic-link login), *every* logged-in user — not just admins — instantly gains full read/update/delete on every table in the system, and insert rights on `job_listings`. This is exactly the kind of gap that's invisible until the day it isn't.

**Rough fix:** introduce a real admin flag (e.g., a `profiles` table with `is_admin boolean`, or Supabase custom claims) and rewrite these policies to check it, instead of trusting the `authenticated` role itself as a proxy for "admin." Until then, admin operations continue through Supabase Studio (service_role, bypasses RLS) as they already do — no functionality is lost by tightening this.

### 1.2 — `job_listings` grants `authenticated` INSERT, not just UPDATE/DELETE (Low, subset of 1.1)

Worth calling out separately because it's the one table where the broad grant would let a non-admin authenticated user *create* fake listings, not just modify/delete curated ones. Same root cause and fix as 1.1.

### 1.3 — `rate_limit_hits` is correctly scoped (informational, no action needed)

Confirmed clean as a positive control: `service_role`-only GRANT, RLS enabled, no policies at all for `anon`/`authenticated`. This is the pattern the other tables should eventually follow for their admin-only operations.

## 2. Storage Bucket Policies

### 2.1 — `company-logos` bucket still has the broad `authenticated` DELETE grant (Low)

`auth_delete_company_logos` (from `20260707000005_create_companies.sql`) has the identical shape to the `auth_delete_cvs` policy just removed from the `cvs` bucket — any authenticated session can delete any company logo. Lower severity than the `cvs` case because logos are re-uploadable, non-sensitive, public-facing images (not private user-submitted documents), but it's the same unenforced assumption and should get the same fix for consistency once 1.1 is addressed.

**Rough fix:** same as the `cvs` fix — drop the policy; route any future admin logo deletion through a service_role-backed path.

### 2.2 — `company-logos` SELECT policy is `anon`-only, not `authenticated` (informational)

`public_read_company_logos` grants SELECT to `anon` but not `authenticated` — meaning an authenticated session currently *cannot* read logos (confirmed empirically while testing 2.1 — a crafted `authenticated` JWT couldn't even see the row via Storage's DELETE-implies-lookup behavior). Not a security issue since the bucket is meant to be public anyway, but it's an inconsistency with the `cvs` bucket's read policy shape (`auth_read_cvs` grants `authenticated`, not `anon`, since `cvs` is private). Worth knowing about if `company-logos` behavior ever needs to be reasoned about via the `authenticated` role.

## 3. API Routes

### 3.1 — `/api/companies/resend-verification` has no rate limiting and no Turnstile check (Low–Medium)

Every other write-adjacent route (`/api/apply`, `/api/post-job`, `/api/subscribe`, `/api/companies/register`, `/api/companies/confirm-logo`) calls `checkRateLimit()` first. `resend-verification` does not, and never had Turnstile either. It does require knowing a real `verification_token` (a UUIDv4, effectively unguessable), so this isn't an open door — but anyone who does have a token (e.g., it leaked via a forwarded email, a shared inbox, or an email-security-scanner log) can trigger unlimited resends, each one a real Resend send and a real DB update. Inconsistent with the rest of the app's posture.

**Rough fix:** add the same `checkRateLimit()` call already used elsewhere (e.g., `resend-verification:${ip}`, a modest limit like 5/hour is consistent with the other routes).

### 3.2 — `getClientIp()` trusts `x-forwarded-for` without validating the proxy chain (Medium)

`lib/rate-limit.ts` takes the first comma-separated value of `x-forwarded-for` at face value. All rate limiting is keyed on this value. On Vercel specifically this header is generally set by Vercel's own edge and is the usual source of truth for client IP — but this wasn't independently re-verified against Vercel's current documented guarantees as part of this audit, and if this app is ever deployed behind anything else (or if Vercel's handling changes), a client sending its own `x-forwarded-for` header could get a fresh rate-limit bucket on every request, fully bypassing the 5/hour (or 3/hour) limits that are the app's main anti-spam/cost-control mechanism.

**Rough fix:** confirm Vercel's current header-handling guarantees explicitly (their docs describe how `x-forwarded-for` is constructed at their edge), and consider also reading Vercel's `x-vercel-forwarded-for` or similar platform-specific header if one exists, as a belt-and-suspenders measure.

### 3.3 — Company registration honeypot is enforced client-side only (Low)

The honeypot field check in `RegisterClient.tsx` (line ~125) runs entirely in the browser before the fetch call — the `RegisterBody` type sent to `/api/companies/register` doesn't include a honeypot field at all, so a script hitting the API directly skips this check entirely. Low severity because the real gates (rate limiting + Turnstile) are already server-side and don't depend on the honeypot; it's currently pure UX/cosmetic filtering for casual browser-based bots, not a security boundary.

**Rough fix:** either drop it (it's not doing real work) or pass the honeypot value through to the API and check it there too, for defense-in-depth against low-effort scripted bots that don't bother solving Turnstile.

### 3.4 — `/api/post-job` and `/api/subscribe` have no Turnstile, rate-limiting only (Low, known/intentional)

Already disclosed as a deliberate lower-priority decision when rate limiting was built ([[rate-limiting-and-bot-protection]]) — restating here so it's visible in the audit trail rather than only in session memory. Still an open gap: a scripted bot can submit up to the rate limit (5/hour and 3/hour respectively) without solving any bot check.

### 3.5 — Rate-limit check fails open on its own DB errors (informational, intentional)

`checkRateLimit()` returns `true` (allow) if the count query itself errors, by explicit design ("fail open rather than blocking legitimate users"). Reasonable trade-off given this table has no bearing on data integrity, but worth knowing that a transient `rate_limit_hits`/service-role connectivity issue silently disables rate limiting for the affected request rather than blocking it.

### 3.6 — Stored HTML injection in outbound email templates (Medium)

`lib/email-templates.ts` interpolates user-controlled fields directly into raw HTML strings with **zero escaping**: `applicant_name`, `job_title`, `company_name`, `why_interested`, `description`, `why_work_here`, `cv_link`, `apply_url`, `website`, `contact_email`, etc. all flow straight into `<td>`, `<p>`, and `href="..."` without any HTML-entity or attribute escaping. Confirmed by reading every template function in the file — every `row()` call and every inline `${...}` in `wrap`/`wrapBase` callers takes raw strings.

Concretely: an applicant or company registrant could submit a value like `"><a href="https://evil.example/phish">Click here to verify your Supabase login</a><div style="` as `applicant_name` or `why_interested`, and the resulting email — sent to `hello@ramenhire.com`, i.e., the person who reviews these — would render the injected link as if it were a legitimate part of the RamenHire notification template, including next to the genuine "Review in Supabase →" button. Most modern email clients block `<script>` execution, so this isn't classic email-based XSS, but it is a real phishing/spoofing vector against whoever reads the admin inbox, and a way to corrupt the visual structure of every notification email.

**Rough fix:** add a small `escapeHtml()` helper (standard `&`, `<`, `>`, `"`, `'` entity replacement) and run every user-supplied string through it before interpolation — including values already wrapped in `<a href="...">` construction, where the escaping needs to cover both the attribute value and the link text.

## 4. Service-role client (`lib/supabase-admin.ts`)

**Checked, no issues found.** Confirmed:
- `SUPABASE_SERVICE_ROLE_KEY` is read without a `NEXT_PUBLIC_` prefix, so Next.js never inlines it into the client bundle.
- Grepped every file importing `@/lib/supabase-admin` (11 matches: all in `app/api/**/route.ts`, `app/companies/register/verify/page.tsx` (a Server Component), and `lib/storage-upload.ts`/`lib/rate-limit.ts`) — none are `"use client"` files, and none of the 8 files that *are* `"use client"` import it, directly or transitively.
- The one Server Component that uses it (`verify/page.tsx`) does the lookup/update server-side only; the page's rendered output never echoes the service-role key or any other secret.

This separation is genuinely sound, not just conventionally assumed — worth stating explicitly since it's the thing that would be catastrophic if wrong.

**Checked, no issues found.** Confirmed:

## 6. Other

### 6.1 — No `middleware.ts` / security headers configured (informational)

No CSP, `X-Frame-Options`/`frame-ancestors`, `Referrer-Policy`, or `Permissions-Policy` are set anywhere in the app. Vercel applies some baseline protections automatically, but this project doesn't opt into any of the stronger ones itself. Not urgent for a pre-revenue MVP with no auth/session cookies to protect, but worth a line item for whenever the site takes on real user sessions.

### 6.2 — `robots.txt` allows everything, including `/api/*` (informational, not a real issue)

`Allow: /` with no disallow rules means `/api/*` routes are technically crawlable. In practice this doesn't leak anything — the API routes are POST-only and return JSON errors to GET requests, and search engines don't meaningfully index JSON — but noting it for completeness since an outside reader might flag it.

## Resolved this session (2026-07-10, follow-up pass)

Same audit cycle as above — the findings below were reviewed and specific items actioned. Everything in this section was implemented **and directly tested locally**, not just code-reviewed.

### Fixed

**2.1 — `company-logos` bucket `authenticated` DELETE grant removed.** New migration `supabase/migrations/20260710000010_revoke_authenticated_delete_company_logos.sql` drops `auth_delete_company_logos`, matching the `cvs` fix. Tested with the same before/after JWT methodology: temporarily restored the old policy, uploaded a fresh test file, and attempted delete as a crafted `authenticated` (non-admin) JWT — **still got 403 even with the old policy present**, because `company-logos` has no `authenticated` SELECT policy at all (only `anon`, via `public_read_company_logos`), and the Storage API's DELETE path requires the caller to be able to see the object first. In other words: this specific grant was already practically inert via the standard Storage REST API before the fix, due to an unrelated pre-existing gap — but the fix is still correct and worth having, since it closes the RLS policy itself (not just an incidental side-effect of a different gap), which matters if `company-logos` ever gains an `authenticated` SELECT policy later, or if a different Storage API code path doesn't have the same pre-check. Final state confirmed via `pg_policies`: only `public_read_company_logos` remains on that bucket.

**3.1 — Rate limiting added to `/api/companies/resend-verification`.** Same `checkRateLimit()` pattern as the other routes, keyed `resend-verification:${ip}`, 5/hour. Tested by sending 6 rapid requests: the first 5 correctly returned `404` (bogus token, expected), the 6th returned `429` with the standard rate-limit message. Test rows cleaned up afterward.

**3.6 — HTML escaping added to all email templates.** New `escapeHtml()` helper in `lib/email-templates.ts` (standard `&`/`<`/`>`/`"`/`'` entity escaping), applied to every user-supplied field across all five template functions (`jobPostTemplate`, `applicationTemplate`, `subscriberTemplate`, `companyRegistrationAdminTemplate`, `companyRegistrationConfirmationTemplate`, `companyVerificationTemplate`) — both plain text content and values used inside `href="..."` attributes. Tested directly against the template functions (bypassing Resend, no real email sent) with two payloads: a `<script>`/`<img onerror=...>` injection in `applicant_name`, and a quote-based attribute-breakout attempt (`" onmouseover="..."`) in `cv_link`. Both rendered as fully inert escaped text in the output HTML — confirmed the raw `<script>` and `onmouseover=` strings never appear unescaped in the generated markup. `npx tsc --noEmit` clean after the change.

**3.3 — Server-side honeypot check added to `/api/companies/register`.** `RegisterClient.tsx` now sends the honeypot field value in the request body; the route checks it before Turnstile verification and silently returns `{ ok: true }` without touching the database if it's filled — mirroring the client's existing "pretend success" behavior so a bot isn't tipped off. Tested directly against the route: a request with the honeypot field filled returned `200 {"ok":true}` and confirmed **zero** rows were written to `companies`; a follow-up request with an empty honeypot and a valid payload completed normally and returned an `upload` URL, confirming the fix doesn't affect real submissions. Test data cleaned up.

### Investigated, then fixed (small, zero-downside change)

**3.2 — `getClientIp()` / `x-forwarded-for`.** Checked Vercel's current docs directly (`vercel.com/docs/headers/request-headers`, last updated 2025-12-13) rather than assuming. Vercel's own text: *"If you are trying to use Vercel behind a proxy, we currently overwrite the X-Forwarded-For header and do not forward external IPs. This restriction is in place to prevent IP spoofing."* — so on a standard Vercel deployment (this app's actual setup, no reverse proxy in front), `x-forwarded-for` is already spoof-resistant and the original concern doesn't hold in practice today.

However, Vercel's docs also describe a second header, `x-vercel-forwarded-for`, as *"identical to the x-forwarded-for header. However, x-forwarded-for could be overwritten if you're using a proxy on top of Vercel"* — i.e. it stays correct even if something like Cloudflare is ever put in front of Vercel later, a scenario where `x-forwarded-for` would no longer be trustworthy. Since preferring it is a zero-downside, one-line change (identical behavior today, strictly more robust against a future infra change), `lib/rate-limit.ts`'s `getClientIp()` now checks `x-vercel-forwarded-for` first, falling back to `x-forwarded-for` then `x-real-ip` exactly as before. Tested end-to-end against a running route: sent requests with a fixed `x-vercel-forwarded-for` value and *varying* `x-forwarded-for` values and confirmed they all shared one rate-limit bucket (got rate-limited together), then sent a request with a *different* `x-vercel-forwarded-for` (same `x-forwarded-for` as the rate-limited one) and confirmed it got a fresh bucket — proving the new header precedence is what's actually driving the rate-limit key, not just present in the code.

### Deliberately deferred, documented (not fixed)

**1.1 / 1.2 — "authenticated = admin" RLS pattern.** Not touched, per explicit instruction — fixing this properly requires real admin-role infrastructure (a `profiles` table with `is_admin`, or custom claims), which is a deliberate future project, not a quick patch. Documented in `CHAT_CONTEXT.md` (§10, item 7) as a known, intentionally deferred item: contained today because no real end-user auth exists in the app yet, and must be addressed before any feature introduces real candidate/employer login.

### Still open (not in scope for this pass)

- **2.2** — `company-logos` SELECT policy inconsistency (informational, no action planned).
- **3.4** — `/api/post-job` and `/api/subscribe` have no Turnstile (known/intentional, rate-limiting only).
- **3.5** — Rate-limit fail-open behavior on DB errors (informational, intentional trade-off).
- **6.1 / 6.2** — No security headers / permissive `robots.txt` (informational, low priority for current MVP phase).
